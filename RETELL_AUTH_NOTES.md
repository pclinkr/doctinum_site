# Retell Web-Call Auth Notes

## What must stay server-side

- `RETELL_API_KEY` must only live on the backend.
- Frontend should only receive short-lived call `accessToken` for `RetellWebClient.startCall`.

## Recommended backend protection

1. Require an authenticated user session (`HttpOnly` cookie or backend auth middleware).
2. Validate origin and enable a strict CORS allowlist.
3. Add per-IP and per-user rate limits on test web-call creation.
4. Optionally require CAPTCHA for public demo pages.
5. Return only non-sensitive data to frontend:
   - `testCallId`
   - `callId`
   - `accessToken`
   - `remainingTests`
6. For your route pattern (`/api/agents/[id]/test-web-call`), verify the `[id]` against an allowlist to avoid arbitrary route abuse.

## If you need token auth from frontend

- Use a short-lived signed token generated by backend (TTL 1-5 min).
- Pass it via `Authorization: Bearer <token>`.
- Never pass the real Retell API key to browser code.
